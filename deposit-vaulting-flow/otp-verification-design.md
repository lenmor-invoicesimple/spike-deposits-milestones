# OTP Verification — Technical Design

> For the buyer-initiated flow (Item #5). Covers security model, implementation, and threat analysis.

---

## Overview

When a buyer opens checkout with a vaulted payment method, we send a fresh 6-digit code to their email. They enter it to prove identity before we reveal/charge the vaulted PM.

---

## Threat Model

An attacker has the public checkout URL and wants to guess/steal the OTP to charge the vaulted payment method.

**Attack vectors:**
1. Brute-force: guess all 1M combinations
2. Interception: read the email in transit
3. DB breach: read stored codes
4. Replay: reuse a previously valid code
5. Timing attack: infer correct digits from response time
6. Flood: request unlimited codes to increase attack surface

---

## Randomness

6-digit code = 1,000,000 possible values (000000–999999).

| Method | Secure? | Why |
|--------|---------|-----|
| `Math.random()` | ❌ No | Pseudo-random, predictable if seed is known |
| `crypto.randomInt(100000, 999999)` | ✅ Yes | Node.js crypto module, uses OS entropy |
| `crypto.randomBytes(3)` → convert | ✅ Yes | Same entropy source |

**Use `crypto.randomInt()`.** One line, no library needed.

---

## Storage — Why Plaintext is Fine

OTP is NOT a password. It's generated by us, not chosen by the user.

| Approach | Effective? | Why |
|----------|-----------|-----|
| Store plaintext | ✅ Acceptable | Code expires in 15 min — if DB is breached, window is tiny |
| Store hashed (SHA-256) | ❌ Security theater | 6-digit code = only 1M possibilities. Attacker brute-forces all hashes in milliseconds |
| bcrypt/scrypt/argon2 | ❌ Overkill | Designed for slow password verification, not ephemeral codes |
| Encryption at rest | Already covered | PG volume-level encryption (if enabled) handles this |

**The real protection is rate limiting + expiry, not cryptographic storage.**

---

## What Actually Protects the OTP

### 1. Rate Limiting (most important)

```
Max 5 attempts per code → code invalidated after 5 wrong guesses
Max 3 codes per email per hour → prevents rapid regeneration
```

Without rate limiting: attacker tries all 1M combinations in seconds → guaranteed breach.
With 5 attempts: 5/1,000,000 = 0.0005% chance of guessing. Effectively zero.

### 2. Short Expiry

- 10-15 minutes. Code becomes useless after that.
- Even if email is intercepted, window is tiny.

### 3. Single-Use

- Once verified (or max attempts hit), code is burned.
- Cannot be replayed.

### 4. Constant-Time Comparison

- Compare using `crypto.timingSafeEqual()` to prevent timing attacks.
- Practically irrelevant over HTTP (network jitter >> timing difference), but free to add and good practice.

### 5. Code Entered, Not in URL

- Code is typed into a form field, not embedded in a clickable link.
- Links get logged (server access logs, browser history, referrer headers). Form input does not.

---

## What We Don't Need

| Technique | Why not |
|-----------|---------|
| Salting | Not a password — we generate it, user doesn't choose it |
| bcrypt/scrypt/argon2 | Brute-force protection comes from rate limiting, not slow hashing |
| HMAC-based OTP (TOTP/HOTP) | For authenticator apps (Google Authenticator). We're sending a code, not generating one client-side |
| Encryption at rest | Volume-level DB encryption covers this. Individual field encryption is overkill for a 15-min-lived value |
| JWT for the code | Overengineering. A DB row with expiry is simpler and revocable |

## Database Schema

```sql
CREATE TABLE checkout_verifications (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  invoice_id    TEXT NOT NULL,
  buyer_email   TEXT NOT NULL,
  code          CHAR(6) NOT NULL,
  attempts      INT NOT NULL DEFAULT 0,
  expires_at    TIMESTAMPTZ NOT NULL,
  verified_at   TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index for lookup on verify
CREATE INDEX idx_checkout_verifications_invoice_email 
  ON checkout_verifications (invoice_id, buyer_email) 
  WHERE verified_at IS NULL;

-- Index for rate limiting (codes per email per hour)
CREATE INDEX idx_checkout_verifications_email_created 
  ON checkout_verifications (buyer_email, created_at);
```

## Implementation

### Generate + Send

```typescript
const generateOTP = async (invoiceId: string, buyerEmail: string) => {
  // Rate limit: max 3 codes per email per hour
  const recentCount = await db('checkout_verifications')
    .where('buyer_email', buyerEmail)
    .where('created_at', '>', db.raw("NOW() - INTERVAL '1 hour'"))
    .count()

  if (recentCount >= 3) {
    throw new TooManyRequestsError('Too many codes requested. Try again later.')
  }

  const code = crypto.randomInt(100000, 999999).toString()

  await db('checkout_verifications').insert({
    invoice_id: invoiceId,
    buyer_email: buyerEmail,
    code,
    expires_at: db.raw("NOW() + INTERVAL '15 minutes'"),
  })

  // Send via is-messages (same pipe as receipt emails)
  await sendOTPEmail(buyerEmail, code, invoiceId)

  return { sent: true }
}
```

### Verify

```typescript
const verifyOTP = async (invoiceId: string, buyerEmail: string, submittedCode: string) => {
  const row = await db('checkout_verifications')
    .where('invoice_id', invoiceId)
    .where('buyer_email', buyerEmail)
    .whereNull('verified_at')
    .orderBy('created_at', 'desc')
    .first()

  if (!row) {
    throw new NotFoundError('No active code. Request a new one.')
  }

  if (row.attempts >= 5) {
    throw new ForbiddenError('Too many attempts. Request a new one.')
  }

  if (new Date(row.expires_at) < new Date()) {
    throw new GoneError('Code expired. Request a new one.')
  }

  // Constant-time comparison
  const codeBuffer = Buffer.from(row.code)
  const submittedBuffer = Buffer.from(submittedCode.padStart(6, '0'))

  if (!crypto.timingSafeEqual(codeBuffer, submittedBuffer)) {
    await db('checkout_verifications')
      .where('id', row.id)
      .increment('attempts', 1)

    const remaining = 5 - (row.attempts + 1)
    throw new UnauthorizedError(`Invalid code. ${remaining} attempts remaining.`)
  }

  // Mark as verified
  await db('checkout_verifications')
    .where('id', row.id)
    .update({ verified_at: db.raw('NOW()') })

  return { verified: true }
}
```

### Session After Verification

Once verified, issue a short-lived session token so the buyer doesn't re-verify on every page load:

```typescript
// After successful verify, return a signed token
const sessionToken = jwt.sign(
  { invoiceId, buyerEmail, verifiedAt: Date.now() },
  process.env.CHECKOUT_SESSION_SECRET,
  { expiresIn: '30m' }
)

// Frontend stores in memory (not localStorage — cleared on tab close)
// Sent as Bearer token on subsequent checkout API calls
```

## Security Summary

| Layer | Implementation |
|-------|---------------|
| Randomness | `crypto.randomInt()` — OS entropy, not Math.random() |
| Brute-force | 5 attempts max per code, then invalidated |
| Flood prevention | Max 3 codes per email per hour |
| Replay protection | Single-use (verified_at set on success) |
| Time window | 15-min expiry |
| Timing attacks | `crypto.timingSafeEqual()` for comparison |
| Transmission | HTTPS (already the case for checkout) |
| Storage | Plaintext — hashing is security theater for 6 digits |
| Session | Short-lived JWT (30 min), memory-only on frontend |

## Upgrade Path

| Phase | Change |
|-------|--------|
| v1 (email OTP only) | As described above — single possession factor, acceptable for US/CA |
| v2 (PIN + email OTP) | Add buyer-chosen PIN at vault time = true 2FA under US standards (GLBA/NIST) |
| v3 (PIN + SMS OTP) | Swap email OTP for SMS OTP — cleaner possession factor, unambiguous EU/SCA compliance |

## PIN + Email OTP (v2) — 2FA Design

Adding a buyer-chosen PIN as a knowledge factor alongside the email OTP (possession factor) satisfies GLBA's MFA definition ([16 CFR 314.2(k)](https://www.law.cornell.edu/cfr/text/16/314.2)):

> "Multi-factor authentication means authentication through verification of at least two of the following types of authentication factors:
> (1) Knowledge factors, such as a password;
> (2) Possession factors, such as a token; or
> (3) Inherence factors, such as biometric characteristics."

| Factor | Our implementation | GLBA category |
|--------|-------------------|---------------|
| PIN | 4-6 digit code buyer creates at vault time | (1) Knowledge — "such as a password" |
| Email OTP | Fresh code sent per session | (2) Possession — "such as a token" |

### PIN Setup (at vault time — deposit payment)

- Buyer pays deposit → card is vaulted
- "Create a 4-digit PIN to secure future payments on this invoice"
- PIN stored **hashed** (bcrypt — user-chosen and reusable, unlike OTP which is plaintext)
- One new column on `vaulted_payment_methods` table

**Why bcrypt (not plaintext):** PIN is user-chosen and reusable. If DB is breached, plaintext PIN could be used to attack the buyer elsewhere. Unlike OTP (system-generated, 15-min lived), PIN needs real password hashing.

**UI label must be clear:** "This is NOT your card PIN" — to prevent buyers accidentally entering their bank card PIN.

### Verification Flow (buyer-initiated checkout)

```
[Checkout opens] → "Enter your 4-digit PIN"
                         ↓ (correct)
              "Code sent to j***@email.com"
              "Enter 6-digit code: [______]"
                         ↓ (correct)
              "Pay $1,500 with AMEX ***123"
              [Pay Now]
```

**PIN-first, then OTP — not simultaneous. Why:**
- PIN gates OTP delivery — wrong PIN = no OTP sent, no attack surface, no email spam
- Attacker with email access alone can't trigger an OTP without knowing the PIN
- Simpler error handling — each screen validates one thing
- Rate limiting is cleaner per step (5 PIN attempts → locked; 5 OTP attempts → locked)

If PIN wrong → "Invalid PIN. X attempts remaining." (OTP never sent)
If OTP wrong → "Invalid code. X attempts remaining." / Resend option

### Forgot PIN Flow

| Option | Security | UX | Recommended? |
|--------|----------|----|--------------|
| Reset via email link | ❌ Collapses 2FA — attacker with email can reset PIN AND get OTP | Easy | ❌ No |
| Force fresh card entry (bypass vault) | ✅ Card entry proves possession | Buyer can still pay | ✅ v1 |
| Reset via SMS | ✅ Separate channel from email | Good | ✅ v2 (needs phone) |
| Merchant resets vault | ✅ Human verification via merchant relationship | Requires merchant action | ✅ Backup option |
| Delay-based reset via email | Weak — just delays the collapse | Terrible (24hr wait) | ❌ No |

**Recommended for v1: Option B (fresh card entry) + merchant reset as backup**

```
[Enter your PIN: ______]
      [Forgot PIN?]
            ↓
"For security, enter your card details to pay"
   [Normal checkout form]
            ↓
   Payment succeeds
            ↓
"Set a new PIN for future payments?" → [____] Save
```

Key insight: "Forgot PIN" doesn't have to mean "reset PIN." It means "bypass vault, pay normally." The vault is a convenience feature — buyer always has the fallback of entering card details fresh.

### Is storing a PIN legal?

Yes — this is an **application PIN**, not a card PIN. Storing it is legal and common practice:
- Not cardholder data (unrelated to their card)
- Not a bank card PIN (PCI-DSS prohibits storing *those*, not application passwords)
- Same as any app storing a hashed user password
- Stored bcrypt-hashed — same standard as password storage everywhere

### GLBA note (US)

PIN + Email OTP satisfies GLBA MFA. However, GLBA only applies to entities that are "financial institutions" under [16 CFR 314.2(h)](https://www.law.cornell.edu/cfr/text/16/314.2). Whether an invoicing SaaS using third-party processors qualifies is legally ambiguous — SaaS platforms aren't mentioned in the regulation (written in 1999). Legal review recommended before making any GLBA compliance claims publicly.

### EU/SCA note

Email as a possession factor is debated under PSD2/SCA — some EU regulators consider it insufficiently separate from how checkout links arrive. For EU-bulletproof compliance: PIN + SMS OTP (v3). The PIN itself carries over unchanged.

## Legal Considerations & Amount Thresholds

### Regulatory Landscape

| Region | Regulation | Applies to us? | What it requires |
|--------|-----------|---------------|-----------------|
| EEA (EU + Norway/Iceland/Liechtenstein) | PSD2/SCA | ❌ Not today (US/CA market) | Two-factor mandatory for electronic payments |
| UK | UK-SCA (post-Brexit equivalent) | ❌ Not today | Same as PSD2, enforced since 2022 |
| India | RBI AFA (Additional Factor Authentication) | ❌ Not today | 2FA for ALL electronic payments, even domestic |
| Australia | AusPayNet CNP Fraud Framework | ❌ Not today | Soft mandate — card networks enforce, not law |
| Japan | Installment Sales Act amendment | ❌ Not today | 3DS required for e-commerce by 2025 |
| US | Reg E | ✅ Yes | Consumer right to dispute unauthorized transfers. Burden on merchant to prove authorization. |
| US/CA | PCI DSS | ✅ Yes | Protects stored card data, not the authorization method. We never store PANs (Stripe does). |
| US/CA | Card network rules (Visa/MC) | ✅ Yes | Liability shift rules for disputes |
| US/CA | State consumer protection laws | ✅ Varies | General "reasonable security" standard. No specific OTP requirement. |

**US and Canada are the outliers** — most major markets are moving toward mandatory strong authentication for payments. If Invoice Simple ever serves merchants in EU, UK, India, or Japan, two-factor authentication becomes a hard legal requirement, not optional.

**For our current US/CA market:** No law mandates a specific OTP channel or strength. The exposure is **liability**, not illegality.

### Chargeback Evidence Strength

When a buyer disputes ("I didn't authorize that"), the card network evaluates our evidence:

| Evidence | Dispute defense strength |
|----------|--------------------------|
| Nothing (bare URL click) | ❌ Near-guaranteed loss |
| Email magic link | Weak — "I didn't click that" / link scanners |
| Email OTP | Moderate — active input, timestamped, IP logged |
| SMS OTP | Strong — separate channel, proves phone possession |
| 3DS (Visa Secure / MC Identity Check) | **Liability shift** — bank absorbs the loss, not us |

**Key point:** Only 3DS shifts liability to the card issuer. All other methods (email OTP, SMS OTP, biometrics) mean we still absorb the chargeback loss if the buyer wins the dispute. OTP strengthens our evidence but doesn't eliminate risk.

### Amount-Based Risk

There's no legal threshold where email OTP becomes "illegal." But risk scales with amount:

| Amount tier | Email OTP risk | Recommendation |
|-------------|----------------|----------------|
| < $500 | Low | Email OTP fine for v1 |
| $500 – $2,000 | Low-Medium | Email OTP acceptable, monitor dispute rate |
| $2,000 – $5,000 | Medium | Consider requiring SMS OTP (v2) |
| $5,000+ | High | Should require SMS + 3DS or stronger |

Why it scales:
- Higher amount = higher incentive to dispute
- Higher amount = more scrutiny from card networks (Visa/MC flag high dispute ratios → higher fees or account termination)
- "Reasonable security" is judged proportional to risk — email OTP for $200 is defensible, for $50K a court might disagree
- Payment fraud insurers may not cover higher tiers with email-only OTP

### What We Must Do (Regardless of Amount)

1. **Log everything** — IP, timestamp, user-agent, email sent time, code entry time, number of attempts. This is chargeback evidence.
2. **Consent language at vault time** — Stripe mandate model: "I authorize [Invoice Simple] to charge my saved payment method for the remaining balance."
3. **Track dispute rate** — if chargebacks on OTP-verified payments exceed 0.5%, escalate to SMS.
4. **Terms of Service** — buyer agrees to stored PM usage at deposit time.

### Optional Safeguards for v1

- **Soft cap with friction** — above $2,000, show confirmation: "You're about to pay $X with AMEX ***123. Confirm?" (adds intent evidence)
- **Amount-based channel escalation** — route to SMS OTP above a threshold once v2 is available
- **3DS for high value** — trigger Stripe 3DS on buyer-initiated charges above $5K (gets liability shift)

### When to Get Legal Review

| Scenario | Legal review needed? |
|----------|---------------------|
| v1 launch, typical invoices ($100–$2K) | No — doing more than most invoice platforms today |
| High-value invoices ($5K+) | Yes — payment compliance opinion before launch |
| EU/UK market expansion | Yes — PSD2/SCA compliance mandatory |
| First chargeback dispute on OTP-verified charge | No — but review evidence package with payments team |
| Dispute rate >0.5% | Yes — operational review + consider escalation to SMS/3DS |

## References — Industry Precedent

Why OTP (not magic links) for authorizing charges on stored payment methods.

### The Distinction

| Use Case | Magic Link OK? | Industry Standard |
|----------|---------------|-------------------|
| Login / site access | ✅ Yes | Common (Notion, Slack, etc.) |
| Password reset | ✅ Yes | Standard practice |
| View account info | ✅ Yes (low-risk) | Acceptable |
| Authorize charge on stored card | ❌ No | OTP / biometrics / 3DS |

**The line:** if the click *triggers money movement* from a stored instrument, magic links are insufficient.

### What Major Players Use

| Provider | Verification for Stored PM | Method |
|----------|---------------------------|--------|
| Stripe Link | OTP | SMS code sent to phone on file |
| Klarna | OTP + app login | SMS/app-based verification |
| Shop Pay | OTP | SMS code |
| PayPal | Full login | Password + optional 2FA |
| Apple Pay | Biometrics | Face ID / Touch ID / device passcode |
| Google Pay | Device unlock | PIN / biometrics |

None use magic links to authorize charges.

### Regulatory & Standards Bodies

#### OWASP — Transaction Authorization Cheat Sheet
`cheatsheetseries.owasp.org/cheatsheets/Transaction_Authorization_Cheat_Sheet.html`

> "Generally, mobile and online applications will require users to submit a second factor so the system can check whether they are authorized to perform a sensitive operation (such as wire transfer authorization)."

Key principle — **"What You See Is What You Sign" (WYSIWYS)**: the user must identify and acknowledge the specific transaction data (amount, recipient). A magic link click carries no transaction-specific context.

OWASP explicitly separates session authentication from transaction authorization — using the same credential for both creates attack surface for replay/MITM.

#### NIST SP 800-63B — Digital Identity Guidelines
`pages.nist.gov/800-63-4/sp800-63b.html`

- **AAL1** (single-factor): acceptable for low-risk access. Magic links fall here.
- **AAL2** (two distinct factors): **required** when personal/financial data is accessible.

> "Federal agencies SHALL select a minimum of AAL2 when personal information is made available online."

> "Proof of the possession and control of two distinct authentication factors is required." (for AAL2)

Magic link = one factor (possession of email inbox). Does not meet AAL2.

#### PSD2 / Strong Customer Authentication (EU)
`docs.stripe.com/strong-customer-authentication`

> "SCA requires authentication to use at least two of the following three elements: Knowledge (something you know), Possession (something you have), Inherence (something you are)."

A magic link is one element (possession). Fails the two-factor requirement for electronic payments. Enforced in EEA, UK, India, Japan, Australia.

> "Banks can decline payments that require 3DS authentication but don't have 3DS enabled."

#### Stripe — Setup Intents (Stored Cards)
`docs.stripe.com/payments/setup-intents`

> "To use merchant-initiated transactions, you need to authenticate the card either when it's first being saved or on the first payment" and must "get an agreement from the customer (mandate)."

Even MIT/off-session charges require strong initial authentication at vault time.

### Why Magic Links Specifically Fail

1. **Single-factor only** — proves email inbox access (possession). Doesn't meet AAL2 or SCA two-factor requirement.
2. **No transaction context** — a click doesn't prove awareness of amount/payee (violates WYSIWYS).
3. **Passive action** — clicking ≠ actively consenting. Weaker chargeback defense.
4. **Links leak** — browser history, referrer headers, server logs, corporate email scanners that pre-fetch URLs.
5. **Forwardable** — can be shared without the recipient knowing the transaction details.
6. **Bot/scanner triggerable** — corporate email gateways and security scanners follow links automatically.

### Why OTP Passes

1. **Active input** — user reads code from one channel, types it in another. Demonstrates presence and intent.
2. **Fresh per-session** — generated only when checkout is opened with a vaulted PM. Narrow time window.
3. **Rate-limitable** — 5 attempts max gives 0.0005% guess probability. Links are binary (valid or not).
4. **Not in URL** — typed into a form field, so it doesn't appear in logs/history/referrers.
5. **Transaction-adjacent** — entered on the payment page itself, proving the user is actively at the point of action.

### Our Position (US/CA Invoice Payments)

- We are not legally bound by PSD2/SCA (US/CA market)
- But we follow the same pattern as Stripe Link (OTP) because:
  - Chargeback defense: active code entry is stronger evidence of cardholder intent
  - Industry norm: no major payment processor uses magic links for stored PM authorization
  - Future-proof: if we expand to EU/UK, we already meet the bar
  - User trust: familiar pattern (same as Stripe Link, Shop Pay, Klarna)
