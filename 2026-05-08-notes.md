# Notes: Payment Scheduling — Implementation Findings

> Scratch notes from spike investigation and Figma review. Not design decisions — just facts worth remembering.

## `invoice.balanceDue` — How It's Calculated

**What it represents:** The amount the client owes *right now* — not the full remaining balance.

**Formula:** `min(nearest upcoming payment amount, remaining balance)`

- If there's an unpaid deposit → returns the deposit amount
- Otherwise → returns the nearest upcoming milestone amount
- Capped at the actual remaining balance (total − all paid amounts)

**Example:** Invoice total $315, no payments made, one upcoming payment of $94.50 → `balanceDue = 94.50` (not $315).

This is what checkout uses as the "Payment Amount" shown to the client.

**Where the formula lives:** `getInvoiceBalance()` in the `@invoice-simple/calculator` npm package (external, source-mapped in `node_modules`).

## EventBridge Schedule Group — What "Reuse" Means

An EventBridge **schedule group** is just a container/folder — it holds individual schedules but has no logic of its own.

Inside `is-payments-retry-schedule-group` (the existing group), you can have schedules with any names:

```
is-payments-retry-schedule-group/
  ├── payment-retry-series123-1        ← existing RP schedule
  ├── payment-retry-series456-2        ← existing RP schedule
  ├── milestone-inv789-ms001           ← new milestone schedule (initial charge)
  └── milestone-retry-inv789-ms001-1   ← new milestone retry schedule
```

RP schedules fire the RP scheduler Lambda. Milestone schedules fire the milestone scheduler Lambda. No interference — they just coexist in the same group under different name prefixes.

"Reuse the group" = milestone schedules are created inside the same existing group. No new CDK group construct needed.

## Paying a Deposit or Milestone Reduces the Balance

Each paid deposit or upcoming payment reduces `balanceDue` — `getInvoiceBalance()` subtracts all payments with `amount + date` set. When the client pays the deposit at checkout, `balanceDue` updates to the next upcoming milestone amount (or full remaining balance if no more milestones).

## `ExtendedCheckoutData.payments[]` — Already Contains All Milestones

`ExtendedCheckoutData.payments[]` includes all Payment objects for the invoice (deposit + all upcoming milestones), fetched via `invoiceGetPayments` in `getDocumentData()` at checkout page load. No extra API call needed to check for upcoming milestones.

This is relevant to the Option B vs E eligibility decision: both options are equally simple at the checkout layer — filtering `payments[]` for upcoming milestones is a trivial array operation on already-loaded data.

## Vaulting Mechanism (Recurring Payments Pattern)

RP uses `PaymentIntent` with `setup_future_usage: 'off_session'` — a single call that charges the deposit and saves the card simultaneously. **Not a separate SetupIntent.** The resulting `vaultedToken` (Stripe `pm_xxx`) is extracted via `getVaultedTokenFromPaymentIntent()` and stored in the `recurring_payment` table.

Milestone auto-pay should follow the same pattern. The `milestone_payment` vault table mirrors `recurring_payment`: `vaultedToken`, `customerId`, `paymentMethodType`, `timezone`, `consentGrantedAt`.

## "Invoice Overdue / Send Reminder" Banner (from Figma Non-Vaulted)

When a milestone's due date has passed and no vault is active, the Invoice Editor shows an orange **"Invoice Overdue — Send Reminder"** banner with a pre-filled overdue reminder email ("This is a friendly reminder that your invoice is overdue...").

**With auto-pay active (vault exists):** this banner/button should be hidden or replaced — the charge is happening automatically, so a manual reminder doesn't make sense. This is a UI detail for Ticket 5b (mobile) to handle: suppress the "Send Reminder" CTA when vault is active for that invoice.

## Automatic Payments Tile — Always Visible on Invoice Editor (Ticket 5a)

The "Invoice Simple PAYMENTS / Automatic Payments" tile is always shown on the Invoice Editor screen — it is not conditionally rendered based on whether milestones exist. Visibility is gated on eligibility only: global feature flag on + provider connected (Stripe and/or PayPal). Within that gate, it shows regardless of invoice state (no deposit, no milestones, no vault). The tile state changes (non-vaulted / vaulted / error) but it's always present for eligible accounts.

Key file: `is-services/packages/services/is-stripe/src/stripe-handle-recurring-payment-confirm.ts`
